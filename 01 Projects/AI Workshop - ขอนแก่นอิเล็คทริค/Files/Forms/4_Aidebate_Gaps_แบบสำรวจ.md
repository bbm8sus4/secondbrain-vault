# Gaps ในแบบสำรวจ AI Workshop — ฉบับสังเคราะห์

หลักการคัด: ใส่เฉพาะ gap ที่ **เปลี่ยนการตัดสินใจจริง** (pacing / แบ่งกลุ่ม / use case / safety net / วัดผล) — ไม่ใช่ถามเผื่อ ทุกข้อระบุชัดว่าใช้ตัดสินใจอะไร

---

## ฟอร์ม 1 — เจ้าของธุรกิจ (เพิ่ม)

| ข้อที่ควรเพิ่ม | ใช้ตัดสินใจอะไร |
|---|---|
| ⭐ **ยอมรับผลลัพธ์แบบ tiered ได้ไหม** — ทุกคนเท่ากัน vs คนเก่งทำ stretch / คนอ่อนทำเวอร์ชันย่อ | **ตัวแปรสำคัญสุดต่อ pacing ทั้งคอร์ส** ตรงกับปัญหาหลัก "ความต่างระดับสูง" ถ้าบังคับเท่ากันหมด ต้องลดเพดานทั้งห้อง / ถ้ายอม → ทำ fast lane + guided lane ได้ |
| **2-3 งานจริงในบริษัทที่อยากให้เว็บไปแก้** | เปลี่ยน generic demo → use case จริง = motivation + วัด ROI ได้ |
| **ใช้ข้อมูลจริงบริษัทได้แค่ไหน** (ลูกค้า/การเงิน/ERP export/PDPA, ต้อง anonymize ไหม) | กำหนดว่าต่อ AI+DB จริงได้แค่ไหน หรือต้องเตรียม dataset ปลอม + guardrail |
| **บังคับหรือสมัครใจ + ผูก KPI/ผลตอบแทนไหม + ให้ฝึกในเวลางานไหม** | ทำนาย drop-off + ตัดสินว่าออกแบบการบ้านนอกคลาสได้ หรือต้องจบในห้อง |
| **หลังจบใครดูแล/จ่าย cloud ต่อ + มี IT internal ไหม** | กันโปรเจกต์ตายหลังคลาส + กำหนด "ความลึกที่ดูแลต่อไหว" |
| **นิยามผลลัพธ์ที่ 30/60 วัน** (prototype ใช้จริง 1 ตัว / ลดเวลา report X% / internal tool deploy แล้ว) | วัดผลจริง ไม่ใช่แค่คะแนนพึงพอใจ (goal 5) |

## ฟอร์ม 2 — ผู้เรียน (เพิ่ม)

| ข้อที่ควรเพิ่ม | ใช้ตัดสินใจอะไร |
|---|---|
| ⭐ **Proxy ตรรกะแทน "เคยโค้ดไหม"** — ใช้สูตร Excel/IF/VLOOKUP, pivot, Zapier/automation, SQL? | ทำนายเร็ว/ช้า **แม่นกว่า** "เคยเขียนโค้ด" — สาย business ที่ใช้สูตรเก่งมักไปไว |
| ⭐ **Micro pre-task ลงมือจริง (แบบเบา→หนัก)** — เปิด terminal, แก้ไฟล์, อ่าน error แล้วให้ Claude ช่วย, *(ออปชัน)* push GitHub / deploy หน้า static | baseline ที่เชื่อได้กว่า self-rate — **ทำให้เบาพอ** ไม่ให้คนไม่เคยโค้ดถอดใจก่อนเริ่ม |
| **ความมั่นใจแยกเป็นทักษะย่อย** — ติดตั้งโปรแกรม / CLI / อ่าน error / GitHub / API key / database | รู้ว่าต้อง scaffold จุดไหนเป็นรายคน |
| **พฤติกรรมเมื่อติด** — ลองเองต่อ / ถาม AI / ถามเพื่อน / รอครู / **หยุดเงียบ** | คัด "คนเสี่ยง drop เงียบ" มาเฝ้าใกล้ชิด |
| **อ่าน error ภาษาอังกฤษเข้าใจไหม + เคยมีประสบการณ์แย่กับการเรียน tech ไหม** | error ทั้งหมดเป็นอังกฤษ = อุปสรรคซ่อนของ non-tech → เตรียม cheat sheet + เฝ้ากลุ่มวิตกกังวล |
| **เคยจัดข้อมูลเป็นหลายตาราง/เชื่อมกันไหม** | ทำนายใครจะงง database schema (เป็น learning objective หลัก) → เตรียม scaffold |
| **โจทย์จริงที่อยากทำ 1 อย่าง** (เว็บอะไร, input/output, ใครใช้, ความถี่) | use case รายคน + เป็น deliverable ไว้วัดผลปลายทาง |
| **อยากเป็น "maker" (สร้างเอง) หรือ "power user" (สั่ง AI ทำงานประจำ)** | ไม่บังคับทุกคนวิ่งเส้นเดียว — ออกแบบ track ตามเจตนา |
| **ยินดีเป็น buddy ไหม / กล้าขอความช่วยเหลือไหม** | จับคู่ให้ถูก (คนเก่งที่ *อยาก* ช่วย + คนอ่อนที่ *กล้า* ถาม) ไม่ฝืนธรรมชาติ |
| **เวลาฝึกนอกคลาส + สเปกเครื่อง/สิทธิ์ติดตั้งวันงาน + learning style** (ดูตัวอย่างก่อน / อธิบาย concept / cheat sheet) | กันปัญหาหน้างาน + วาง scaffolding หลายแบบในห้องเดียว + ตั้งการบ้านให้สมจริง |

## ฟอร์ม 3 — ควรเพิ่ม: **หัวหน้าแผนก (รายแผนก)**

**จำเป็น** — ผู้เรียนมักไม่รู้ว่า "งานไหนคุ้มทำเป็นเว็บ" และเจ้าของมองภาพรวมเกินกว่าจะรู้ pain รายแผนก

| ข้อ | ใช้ตัดสินใจอะไร |
|---|---|
| **2-3 งานซ้ำๆ ที่กินเวลาทีมมากสุด + ไฟล์/report/form ตัวอย่างจริง** | คัด use case ที่มี ROI จริง + ทำ workshop asset ให้เหมือนงานจริง |
| **ในทีมใครเหมาะเป็น champion / พี่เลี้ยง tech** | ต่อยอด buddy ข้ามวันอบรม → มี support หลังจบ |
| **ความเสี่ยงถ้าทำพลาด** (ตัวเลขการเงิน / ข้อมูลลูกค้า / production / หน้างาน) | สอน guardrail + approval flow + data policy + rollback ให้ตรงจุด |
| **ยินดีให้ลูกน้องเอาผลงานไปใช้จริง + ให้เวลาทำ mini sprint/clinic + รีวิว 30/60 วันไหม** | นี่คือ **"ตัววัดผลจริง" (goal 5)** — ผูกชิ้นงานกับการใช้งาน ไม่ใช่แค่ความพึงพอใจ |

---

## ช่องโหว่ "การวัดผล" (ครอบทุกฟอร์ม — goal 5)

ทั้ง 2 ฟอร์มปัจจุบันวัดแค่ input + ความพึงพอใจ ยังขาด:
1. **Baseline จาก micro pre-task** เทียบกับผลปลายทาง (วัด progress จริง)
2. **Deliverable รูปธรรมต่อคน + follow-up 30/60 วัน** ว่าเอาไปใช้จริงไหม
3. ตัวชี้วัด = **"ทำอะไรได้เองโดยไม่ต้องช่วย"** ไม่ใช่คะแนนความชอบ

## 3 Insight ที่สำคัญสุด

- **ถามเจ้าของ "ยอมรับ tiered outcome ได้ไหม"** — ปลดล็อกการออกแบบ pacing ทั้งคอร์ส ตอบตรงปัญหาหลัก "ความต่างระดับสูง" ถ้าไม่ถามข้อนี้ จะวาง track เร็ว/ช้าไม่ได้เลย
- **ใช้ Excel/automation proxy + micro pre-task (แบบเบา)** แทนคำถาม "เคยโค้ดไหม" → แบ่งเร็ว/ช้าแม่นกว่ามาก โดยไม่ทำให้คนไม่เคยโค้ดถอดใจ
- **เพิ่มฟอร์มหัวหน้าแผนก** ได้ทั้ง use case จริง + กลไกวัดผลหลังอบรมในชอตเดียว — เป็นจุดที่ทั้งเจ้าของและผู้เรียนมองไม่เห็น
