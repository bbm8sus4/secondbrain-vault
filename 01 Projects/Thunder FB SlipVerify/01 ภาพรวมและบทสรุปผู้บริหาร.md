# 📄 ภาพรวมและบทสรุปผู้บริหาร

> กลับไป [[00 แผนที่ความรู้ (MOC)]]

## เราจะทำอะไร
บอท **ตรวจสลิปโอนเงินอัตโนมัติในช่องทาง Facebook** — ลูกค้าส่งรูปสลิปในแชทเพจร้าน บอทตรวจว่า **จริง / ปลอม / ซ้ำ / ยอดตรงไหม / บัญชีถูกไหม** แล้วตอบใน **2–5 วินาที**

**ต่อยอดจากของที่มี:** ใช้ [[06 EasySlip และ Thunder API|EasySlip / Thunder Slip Verification API]] เดิม — แค่ขยายช่องทางจาก LINE → Facebook (ไม่สร้าง engine ใหม่)

## บทสรุป: ทำได้จริง และคุ้มที่จะทำ
แต่ต้องเลือกช่องทางให้ถูก เพราะ Facebook ไม่เหมือน LINE → เริ่มที่ **Messenger (แชท)** ก่อน ดู [[11 แนวทางเลือก A-D]]

## 🎯 ประเด็นชี้ขาด 4 ข้อ
1. **Facebook อนุญาตให้ทำระบบยืนยันการชำระเงินได้** — Meta เขียนเองว่าสิทธิ์ `pages_messaging` ใช้ทำ *"transactional confirmations"* ได้ ไม่ผิดนโยบาย
2. **EasySlip API ต่อ Facebook ได้เกือบ plug-and-play** — endpoint `verify/bank` รับ **URL รูปตรงๆ** เอา URL จาก webhook ยิงเข้าไปได้เลย มี `checkDuplicate`/`matchAmount`/`matchAccount` ครบ
3. **Thunder ยังไม่มีของบน Facebook เลย แต่คู่แข่งมีแล้ว** — ZWIZ.AI, Chatcone, Page365, Commerzy ทำไปแล้ว ช้ากว่านี้ = เสียพื้นที่ (ดู [[07 คู่แข่งในตลาด]])
4. **ตรงกับโจทย์ pivot ของ Thunder BOT พอดี** — BOT บน LINE กำลังทรงตัว/ถดถอย, การขยายไป Facebook = เอา engine เดิมเปิดตลาดใหม่ โดยให้ Thunder API (ตัวโตเร็ว) เป็นเครื่องยนต์ทำเงินข้างหลัง (ดู [[10 ข้อมูลบริษัท Thunder Group]])

## 🔴 ประเด็นที่ต้องเคลียร์ก่อนสเกล
**KBank API ทางตรงห้ามร้านบนแพลตฟอร์ม Social Media (Facebook/Line)** ขึ้นทะเบียน + รับเฉพาะนิติบุคคล → นี่คือเหตุผลที่ตัวกลาง (EasySlip/SlipOK) มีอยู่ เราต้องต่อผ่าน **EasySlip/Thunder API เท่านั้น** และเช็คว่าเงื่อนไขบริการรองรับ use case "บน Facebook"
→ รายละเอียดใน [[09 กฎหมาย ความเสี่ยง และคำถามเปิด]]

## 💡 ข้อเสนอแนะ
เดิน **แนวทาง A → B** เป็นเฟส: เริ่ม Messenger bot (แชท) ให้ผ่านและเห็นผลจริงก่อน แล้วค่อยต่อยอดคอมเมนต์/ไลฟ์ · ทำ **บน EasySlip/Thunder API ที่มีอยู่** · เริ่มแบบ **Pilot เพจเดียว** ก่อน (ดู [[03 แผนพัฒนา 3 สเตจ และ Timeline]])
