# 🔄 Flow การทำงาน (end-to-end)

> กลับไป [[00 แผนที่ความรู้ (MOC)]] · อ่านคู่กับ [[03 แผนพัฒนา 3 สเตจ และ Timeline]]

## 1. ร้านค้าเริ่มใช้ (onboarding)
```
ร้านสมัคร → เชื่อมเพจ Facebook ของร้านเข้าระบบ → ตั้งบัญชีปลายทาง (เลขบัญชี/พร้อมเพย์ของร้าน)
→ เปิดใช้งาน → บอทเริ่มเฝ้าแชทเพจอัตโนมัติ
```
_(ใน Pilot: ข้ามขั้นนี้ — ตั้งเพจของเราเองให้เสร็จจากหลังบ้าน)_

## 2. เส้นทางปกติ (happy path)
```
ลูกค้าทักแชทเพจร้าน → โอนเงิน → ส่งรูปสลิปในแชท
   → บอทเห็นรูปทันที → ตรวจ → ตอบใน 2–5 วิ:
      "✅ ยืนยันแล้ว: โอน 1,500 บาท เข้าบัญชีร้าน เวลา 14:30 — ขอบคุณค่ะ"
   → (ถ้าเชื่อมออเดอร์) อัปเดตสถานะเป็น "รอจัดส่ง" อัตโนมัติ
```

## 3. เคสที่ต้องรับมือให้ครบ — หัวใจว่าเชื่อถือได้ไหม
| สถานการณ์ | บอทต้องตอบ / ทำ |
|-----------|----------------|
| สลิปปลอม/แก้ตัวเลข | ❌ "ตรวจไม่พบธุรกรรมนี้ กรุณาส่งสลิปจริง" + ไม่อัปเดตออเดอร์ |
| สลิปซ้ำ (เอาของเก่ามาใช้) | ⚠️ "สลิปนี้ถูกใช้ไปแล้ว" (จับด้วย `checkDuplicate`) |
| ยอดไม่ตรงออเดอร์ | ⚠️ "ยอดโอน 1,000 แต่ยอดสั่ง 1,500 — ขาด 500" (`matchAmount`) |
| โอนผิดบัญชี | ⚠️ "บัญชีปลายทางไม่ใช่ของร้าน" (`matchAccount`) |
| รูปเบลอ/ไม่มี QR | 🔁 "อ่านสลิปไม่ออก ส่งรูปที่ชัดขึ้นได้ไหมคะ" |
| สลิปเก่าเกินกำหนด | ⚠️ "สลิปเกินอายุที่ตรวจได้" (KBANK 180 วัน, SCB 90 วัน ฯลฯ) |
| ลูกค้ากลับมาหลัง 24 ชม. | ตอบด้วย Utility Message (ต้องเตรียม template) |
| ธนาคาร/ระบบล่ม | 🔁 "ระบบธนาคารขัดข้อง กำลังลองใหม่" + retry อัตโนมัติ |

> 💡 **ความสำเร็จอยู่ที่การจัดการเคส "ไม่ happy" เหล่านี้ให้ครบ** — ไม่ใช่แค่ happy path

## 4. Flow เทคนิค (end-to-end)
```
[ลูกค้า] --ส่งรูปสลิปในแชท--> [Meta/เพจร้าน] --webhook "messages" (มี url รูป)-->
   [ระบบเรา]
     1. Webhook Receiver — verify signature, ตอบ Meta 200 ใน 5 วิ, รู้ว่าเพจไหน
     2. โยนงานเข้า queue (แยกงานหนักออกจาก webhook)
     3. Verification Worker — ยิง EasySlip: POST /v2/verify/bank { url, checkDuplicate, matchAmount, matchAccount }
     4. ตีความผล → ตอบลูกค้าในแชท (Send API)
     5. หักโควตา + เก็บ log (ไม่เก็บรูปถาวร)
```

## จุดสำคัญ 3 จุด (พลาดไม่ได้)
1. ⏱️ **ต้องตอบ Meta 200 ใน 5 วินาที** → งานหนัก (เรียก EasySlip) ต้องแยกไป worker/queue ห้ามทำค้างใน webhook ไม่งั้น Meta ตัด subscription
2. 🔗 **EasySlip รับ URL รูปตรงๆ** → ส่ง url จาก webhook เข้าไปได้เลย ไม่ต้องโหลดรูปมาเก็บ (ดู [[06 EasySlip และ Thunder API]])
3. 🔒 **ไม่เก็บรูปสลิปถาวร** → เก็บแค่ผลลัพธ์/metadata (ตาม Meta Platform Terms + PDPA — ดู [[09 กฎหมาย ความเสี่ยง และคำถามเปิด]])

## องค์ประกอบระบบ (Stage 2 เต็ม — Pilot ใช้แค่ 2,3,4 + EasySlip)
| # | Component | หน้าที่ |
|---|-----------|--------|
| 1 | Merchant Onboarding (OAuth) | ร้านล็อกอิน FB, เก็บ+รีเฟรช Page Token, subscribe เพจ |
| 2 | Webhook Receiver | รับ event ทุกเพจที่ 1 endpoint, verify signature, ตอบ 200 เร็ว, push เข้า queue |
| 3 | Verification Worker | เรียก EasySlip, ตีความผล, retry เมื่อ bank API ล่ม |
| 4 | Reply / Notify | ตอบในแชทด้วย Send API (24 ชม.); เผื่อ Utility Message นอกกรอบ |
| 5 | Merchant Dashboard | ร้านเชื่อมเพจ, ตั้งบัญชีปลายทาง, ดูประวัติ+โควตา |
| 6 | Billing/Quota | หักโควตาต่อการตรวจ, reuse pre-paid เดิม |
| 7 | Data handler | pass-through รูป (ไม่เก็บถาวร), เก็บเฉพาะ metadata, ลบตามกำหนด |
