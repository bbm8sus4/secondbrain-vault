# 📋 สิ่งที่ต้องเตรียม (Checklist)

> กลับไป [[00 แผนที่ความรู้ (MOC)]] · อ่านคู่กับ [[03 แผนพัฒนา 3 สเตจ และ Timeline]]

## 👥 คน
- [ ] **Dev backend** — ทำ webhook + ต่อ EasySlip + logic ตอบ
- [ ] **คนดูแลฝั่ง Meta/กฎหมาย** — เอกสารยืนยันธุรกิจ + App Review + PDPA

## 🔑 บัญชี / สิทธิ์
- [ ] Facebook Developer account + Business Manager
- [ ] Facebook App (เพิ่ม product "Messenger")
- [ ] เพจ Facebook สำหรับ pilot (ของเราเอง)
- [ ] Server ที่มี **HTTPS/SSL จริง** (self-signed ใช้ไม่ได้)
- [ ] **EasySlip API key + โควตา** สำหรับ dev ทดสอบ (+ whitelist IP ของ server)

## 📄 เอกสาร (สำหรับ Stage 2 App Review)
- [ ] หนังสือรับรองบริษัท Thunder (**Business Verification**)
- [ ] **Privacy Policy** รองรับข้อมูลสลิป (PDPA)
- [ ] **วิดีโอ demo use-case** สำหรับ App Review
- [ ] แอปที่ทดสอบได้จริงบนเพจ dev (ใช้ตอน review)

## สิทธิ์ (permissions) ที่ต้องขอตอน Stage 2
- **Messenger:** `pages_show_list`, `pages_manage_metadata`, `pages_messaging`, `pages_read_engagement`, `business_management`
- **Instagram (Stage 3):** เพิ่ม `instagram_basic` + `instagram_manage_messages`
- **ส่งนอก 24 ชม. (optional):** `page_utility_messaging`
- ชุดขั้นต่ำแค่รับ-ส่งได้ = `pages_manage_metadata` + `pages_read_engagement` + `pages_messaging`
- รายละเอียดสิทธิ์/นโยบาย: [[05 ช่องทาง Facebook และนโยบาย Meta]]

## ⚠️ เคลียร์ก่อนสเกล (บล็อกเกอร์)
- [ ] **ยืนยันฐานกฎหมาย** — EasySlip/ITMX รองรับ use case บน Facebook (KBank ห้ามร้าน social media ต่อ API ตรง — เราเป็นตัวกลาง แต่ต้องมั่นใจ) → [[09 กฎหมาย ความเสี่ยง และคำถามเปิด]]

## Assumption ที่ตั้งไว้ (ให้ทีมยืนยัน)
1. reuse EasySlip/Thunder API + key/โควตาที่มีอยู่ — ไม่สร้าง engine ใหม่
2. มีทีม dev พร้อมรับแผนระดับสถาปัตยกรรมไปทำต่อ
3. Pilot ใช้เพจของเราเอง 1 เพจ
4. นิติบุคคล Thunder เป็นผู้ยื่น App Review ตอน Stage 2
