# 🗺️ ทำโปรเจคตรวจสลิปใน Facebook ให้สำเร็จ — ภาพรวมทั้งหมด

> เอกสารนี้ตอบคำถาม *"ต้องทำอะไรบ้าง + flow เป็นยังไง ถึงจะสำเร็จ"* แบบครบ end-to-end
> อ่านเพื่อ **เข้าใจภาพ** ก่อนตัดสินใจลงมือ — ไม่ใช่คำสั่งให้เริ่มทำ

---

## 0. "สำเร็จ" หมายความว่าอะไร (ตั้งเป้าให้ตรงกันก่อน)

โปรเจคนี้สำเร็จเมื่อ: **ลูกค้าปลายทางส่งรูปสลิปในช่องทาง Facebook ของร้าน → ระบบตรวจอัตโนมัติว่าจริง/ปลอม/ซ้ำ/ยอดตรงไหม → ตอบกลับถูกต้องภายในไม่กี่วินาที → ร้านเชื่อถือได้จนใช้แทนการเช็คมือ**

วัดได้ 3 ระดับ:
- **Pilot สำเร็จ:** flow ทำงานจริงบนเพจเราเอง ตรวจแม่นทุกเคสทดสอบ
- **Product สำเร็จ:** ร้านค้าจริงเชื่อมเพจตัวเองแล้วใช้งานประจำ (retention)
- **ธุรกิจสำเร็จ:** ลูกค้า Facebook เติบโตจนเป็นช่องทางรายได้ใหม่ของ Thunder (เสริมโจทย์ pivot จาก LINE)

---

## 1. Flow ฝั่งผู้ใช้จริง (ต้องเห็นภาพนี้ก่อนคิดเทคนิค)

### 1.1 ร้านค้าเริ่มใช้ (onboarding)
```
ร้านสมัคร → เชื่อมเพจ Facebook ของร้านเข้าระบบ → ตั้งบัญชีปลายทาง (เลขบัญชี/พร้อมเพย์ของร้าน)
→ เปิดใช้งาน → บอทเริ่มเฝ้าแชทเพจอัตโนมัติ
```
_(ใน Pilot: ข้ามขั้นนี้ — เราตั้งเพจของเราเองให้เสร็จจากหลังบ้าน)_

### 1.2 เส้นทางปกติ (happy path)
```
ลูกค้าทักแชทเพจร้าน → โอนเงิน → ส่งรูปสลิปในแชท
   → บอทเห็นรูปทันที → ตรวจ → ตอบใน 2-5 วิ:
      "✅ ยืนยันแล้ว: โอน 1,500 บาท เข้าบัญชีร้าน เวลา 14:30 — ขอบคุณค่ะ"
   → (ถ้าเชื่อมออเดอร์) อัปเดตสถานะเป็น "รอจัดส่ง" อัตโนมัติ
```

### 1.3 เคสที่ต้องจัดการให้ครบ (นี่คือหัวใจว่าจะเชื่อถือได้ไหม)
| สถานการณ์ | บอทต้องตอบ/ทำ |
|-----------|----------------|
| สลิปปลอม/แก้ตัวเลข | ❌ "ตรวจไม่พบธุรกรรมนี้ กรุณาส่งสลิปจริง" + ไม่อัปเดตออเดอร์ |
| สลิปซ้ำ (เอาของเก่ามาใช้) | ⚠️ "สลิปนี้ถูกใช้ไปแล้ว" (จับได้ด้วย `checkDuplicate`) |
| ยอดไม่ตรงออเดอร์ | ⚠️ "ยอดโอน 1,000 แต่ยอดสั่ง 1,500 — ขาด 500" (`matchAmount`) |
| โอนผิดบัญชี | ⚠️ "บัญชีปลายทางไม่ใช่ของร้าน" (`matchAccount`) |
| รูปเบลอ/ไม่มี QR | 🔁 "อ่านสลิปไม่ออก ส่งรูปที่ชัดขึ้นได้ไหมคะ" |
| สลิปเก่าเกินกำหนด | ⚠️ "สลิปเกินอายุที่ตรวจได้" (KBANK 180 วัน, SCB 90 วัน ฯลฯ) |
| ลูกค้ากลับมาหลัง 24 ชม. | ตอบด้วย Utility Message (ต้องเตรียม template) |
| ธนาคาร/ระบบล่ม | 🔁 "ระบบธนาคารขัดข้อง กำลังลองใหม่" + retry อัตโนมัติ |

💡 **ความสำเร็จอยู่ที่การจัดการเคส "ไม่ happy" เหล่านี้ให้ครบ** — ไม่ใช่แค่ happy path

## 2. Flow เทคนิค end-to-end

```
┌─ ลูกค้า ─┐        ┌─ Meta/Facebook ─┐      ┌──── ระบบเรา (backend) ────┐     ┌─ EasySlip API ─┐
│ ส่งรูป   │──แชท──▶│ เพจร้าน          │      │                           │     │                │
│ สลิป     │        │ ยิง webhook      │─────▶│ 1. Webhook Receiver       │     │                │
└──────────┘        │ (event=messages, │      │    - verify signature     │     │                │
                    │  มี url รูป)     │      │    - ตอบ 200 ใน 5 วิ      │     │                │
                    └──────────────────┘      │    - แยกว่าเพจร้านไหน      │     │                │
                                              │    - โยนเข้า queue         │     │                │
                                              │ 2. Worker ดึงงาน          │─────▶│ POST /verify/  │
                                              │    - ส่ง url รูป + options │     │ bank           │
                                              │                           │◀────│ คืนผล (ยอด/    │
                                              │ 3. ตีความผล               │     │ บัญชี/ซ้ำ/...)  │
                                              │ 4. ตอบลูกค้า (Send API)   │     └────────────────┘
                    ┌──────────────────┐      │ 5. หักโควตา + log         │
                    │ เพจร้าน          │◀─────│    (ไม่เก็บรูปถาวร)        │
                    │ ส่งข้อความตอบ    │      └───────────────────────────┘
                    └────────┬─────────┘
┌──────────┐                 │
│ ลูกค้า   │◀────แชท─────────┘
│ เห็นผล   │
└──────────┘
```

**จุดสำคัญ 3 จุด:**
1. **ต้องตอบ Meta 200 ใน 5 วิ** → งานหนัก (เรียก EasySlip) ต้องแยกไป worker/queue ห้ามทำค้างใน webhook
2. **EasySlip รับ URL รูปตรงๆ** → ส่ง url จาก webhook เข้าไปได้เลย ไม่ต้องโหลดรูปมาเก็บ
3. **ไม่เก็บรูปสลิปถาวร** → เก็บแค่ผลลัพธ์/metadata (ตาม Meta Platform Terms + PDPA)

## 3. วิธีการทำ ทีละขั้น (จาก 0 → เปิดใช้จริง)

### 🟢 กลุ่มงานที่เริ่มได้ทันที (Pilot — ไม่ต้องรอ Meta)
- **Step 1 — ตั้ง Facebook App:** สร้าง App ใน developers.facebook.com, เพิ่ม product "Messenger", ผูกเพจของเราเอง, เอา Page Access Token
- **Step 2 — Webhook:** ทำ endpoint HTTPS (ต้องมี SSL จริง), ตั้ง verify token, subscribe field `messages`, ตอบ challenge + 200 ใน 5 วิ
- **Step 3 — ต่อ EasySlip:** เอา API key เดิม, เรียก `POST /v2/verify/bank` ด้วย `url` ของรูป + `checkDuplicate`/`matchAmount`/`matchAccount`
- **Step 4 — Logic + ตอบ:** ตีความผลเป็นข้อความ (ตามตาราง 1.3), ตอบด้วย Send API
- **Step 5 — ทดสอบ:** ส่งสลิปจริง 10-20 ใบ (จริง/ปลอม/ซ้ำ/ยอดผิด) → ตรวจว่าตอบถูกทุกเคส
→ **จบ Pilot: พิสูจน์ว่า flow ใช้ได้จริง**

### 🟡 กลุ่มงานตอนขยายเป็น Product (Stage 2 — ต้องรอ Meta)
- **Step 6 — Onboarding ร้าน (OAuth):** ทำ Facebook Login ให้ร้านเชื่อมเพจตัวเอง, เก็บ+รีเฟรช token หลายเพจ, subscribe อัตโนมัติ
- **Step 7 — Business Verification:** ยืนยันตัวตนบริษัท Thunder กับ Meta _(เริ่มเนิ่นๆ — รออนุมัตินาน)_
- **Step 8 — App Review:** ขอ Advanced Access สิทธิ์ `pages_messaging`, `pages_manage_metadata`, `pages_show_list`, `pages_read_engagement`, `business_management` — ต้องมีวิดีโอ use-case + privacy policy + แอปที่ demo ได้
- **Step 9 — Dashboard + Billing:** หน้าจอให้ร้านเชื่อมเพจ/ดูประวัติ/โควตา + ผูกระบบ pre-paid เดิม
- **Step 10 — Closed beta → Launch:** ร้านจริง 2-3 ร้านก่อน แล้วเปิดตัว

### 🔵 กลุ่มงานต่อยอด (Stage 3)
- **Step 11 — คอมเมนต์/ไลฟ์:** subscribe `feed`, ดึงรูปคอมเมนต์ (เรียก Graph API เพิ่ม), Private Reply เด้งเข้าแชท
- **Step 12 — Instagram DM:** reuse core, เพิ่มสิทธิ์ IG

## 5. ปัจจัยที่ทำให้ "สำเร็จ"
1. **ความแม่น + ครบเคส** — จับปลอม/ซ้ำ/ยอดผิดได้จริง (ไม่ใช่แค่ happy path) → ร้านถึงกล้าเลิกเช็คมือ
2. **เร็ว** — ตอบใน 2-5 วิ (EasySlip เร็วพออยู่แล้ว, อยู่ที่ pipeline เรา)
3. **เริ่มเล็ก พิสูจน์ก่อน** — pilot เพจเดียวให้เห็นผลจริง ก่อนลงทุนทำ product เต็ม
4. **ยื่น Meta review เนิ่นๆ** — เป็นคอขวด อย่ารอจนท้าย
5. **ต่อยอดจากของที่มี** — reuse EasySlip API + billing + ฐานลูกค้า Thunder ไม่สร้างใหม่
6. **ประสบการณ์ในแชทลื่น** — ข้อความตอบเป็นภาษาคน ชัด ไม่กำกวม

---

## 6. กับดักที่ทำให้ "ล้ม" (ต้องเลี่ยง)
- 🔴 **เจอบล็อกเรื่องนโยบายธนาคารตอนสเกล** — ถ้าไม่เคลียร์ฐานกฎหมาย FB ก่อน อาจโดนตัดตอนโตแล้ว → เคลียร์ตั้งแต่ต้น
- 🔴 **ไม่ผ่าน App Review** — เตรียมเอกสาร/วิดีโอไม่ครบ → ยื่นเนิ่นๆ เผื่อแก้หลายรอบ
- 🔴 **webhook ตอบช้าเกิน 5 วิ** → Meta ตัด subscription → ต้องใช้ queue แยกงานหนัก
- 🔴 **ทำ product เต็มก่อนพิสูจน์ flow** → ลงทุนเยอะแล้วพบว่าเคสจริงไม่เวิร์ค → pilot ก่อนเสมอ
- 🔴 **เก็บรูปสลิปถาวร** → ผิด Meta Platform Terms + PDPA → pass-through เท่านั้น
- 🟡 **CDN ย่อรูปคอมเมนต์** (Stage 3) → อ่าน QR ไม่ออก → ต้องเทสต์/ดึงไฟล์ต้นฉบับ

## 7. Dependencies & ความเสี่ยงที่คุมไม่ได้ (ต้องรู้ว่ามี)
- **Meta เปลี่ยนกฎบ่อย** — เช่น message tag เพิ่งถูกยกเลิก เม.ย. 2026 → ออกแบบให้ยืดหยุ่น
- **พึ่ง EasySlip → ITMX → ธนาคาร** — ถ้าต้นน้ำล่ม/เปลี่ยน API บริการกระทบ (retry + แจ้งผู้ใช้ให้ดี)
- **นโยบายธนาคารเรื่อง social media** — จุดที่ต้องเฝ้า (ดูข้อ 6)

---

## 8. ตัวชี้วัดว่าสำเร็จ (ดูตอนมีผู้ใช้จริง)
- อัตราตรวจสำเร็จ/ตรวจไม่ออก (ยิ่งอ่านออกเยอะยิ่งดี)
- เวลาตอบเฉลี่ย (เป้า < 5 วิ)
- จำนวนสลิปปลอm/ซ้ำที่จับได้ (มูลค่าที่ป้องกันการโกง)
- ร้านที่ยังใช้ต่อเนื่อง (retention) + จำนวนสลิป/เดือน
