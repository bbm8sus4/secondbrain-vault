# 📄 ระบบตรวจสลิปใน Facebook — สรุปเพื่อการตัดสินใจ

> เอกสารนี้แยกชัดเจนว่าข้อมูลไหน **[หามาเพิ่ม]** จากแหล่งสาธารณะ และไหน **[จากไฟล์บริบท]** ที่ Bobsanchez ส่งมา
> ⚠️ ข้อมูลนโยบาย Meta และราคา อ้างอิง ณ ก.ค. 2026 — ควร verify ซ้ำก่อนลงมือจริง เพราะ Meta เปลี่ยนกฎบ่อย

---

> 🔎 **ระดับความเชื่อมั่นของข้อมูล:** ส่วน Facebook/Meta (ข้อ 2-3) มาจาก **เอกสารทางการ Meta ล้วน + ผ่านการตรวจสอบไขว้ 3 เสียง** (verify แล้ว 23 ข้อ, หักล้างทิ้ง 2 ข้อ). ส่วน EasySlip/คู่แข่ง/บริษัท (ข้อ 4-6) มาจาก **เอกสาร/เว็บทางการของผู้ให้บริการเอง** (first-party) แต่ยังไม่ได้ตรวจสอบไขว้เชิงลึก — เชื่อถือได้ระดับหนึ่ง ควรยืนยันตัวเลขซ้ำก่อนตัดสินใจลงทุน

## 1. บทสรุปผู้บริหาร (อ่านหน้าเดียวจบ)

**ทำได้จริง และคุ้มที่จะทำ** — แต่ต้องเลือกช่องทางให้ถูก เพราะ Facebook ไม่เหมือน LINE

🎯 **ประเด็นชี้ขาด 4 ข้อ:**

1. **Facebook อนุญาตให้ทำระบบยืนยันการชำระเงินได้** — Meta เขียนเองว่าสิทธิ์ `pages_messaging` ใช้ทำ *"transactional confirmations"* ได้ ไม่ผิดนโยบาย

2. **EasySlip API ต่อกับ Facebook ได้เกือบ plug-and-play** — endpoint `verify/bank` รับ **URL รูปตรงๆ** เอา URL รูปสลิปที่ Messenger ส่งมาใน webhook ยิงเข้าไปได้เลย มี `checkDuplicate`/`matchAmount`/`matchAccount` ครบ

3. **Thunder ยังไม่มีของบน Facebook เลย แต่คู่แข่งมีแล้ว** — ZWIZ.AI, Chatcone, Page365, Commerzy ทำตรวจสลิปบน Facebook ไปแล้ว ช้ากว่านี้ = เสียพื้นที่

4. **งานนี้ตรงกับโจทย์ pivot ของ Thunder BOT พอดี** *(จากไฟล์บริบท)* — BOT บน LINE กำลังทรงตัว/ถดถอย การขยายไป Facebook คือการเอา engine ที่มีอยู่ไปเปิดตลาดใหม่ โดยให้ Thunder API (ตัวโตเร็ว) เป็นเครื่องยนต์ทำเงินข้างหลัง

🔴 **ประเด็นที่ต้องเคลียร์ก่อนสเกล (สำคัญมาก — verify แล้ว):** **KBank API ทางตรง "ห้าม" ร้านค้าที่อยู่บนแพลตฟอร์ม Social Media (Facebook/Line)** ขึ้นทะเบียนใช้บริการ (อยู่ในลิสต์ "ธุรกิจที่ธนาคารไม่อนุญาต") และรับเฉพาะนิติบุคคล — **นี่คือเหตุผลว่าทำไมตัวกลางอย่าง EasySlip/SlipOK ถึงมีอยู่** (ไปต่อ ITMX/ธนาคารแทนร้านค้า). แปลว่าเราต้องต่อผ่าน **EasySlip/Thunder API เท่านั้น ห้ามให้ร้านไปต่อ bank API ตรง** และต้องเช็คให้ชัดว่าเงื่อนไขบริการของ EasySlip เองรองรับ use case "บน Facebook" _(เคยมีดีล KBank×Meta ปี 2566 ให้เช็คสลิปใน Messenger ได้ แต่นั่นเป็น integration ตรงของ Meta+KBank คนละเรื่องกับร้านมาสมัคร API เอง)_

💡 **ข้อเสนอแนะ:** เริ่มที่ **Messenger bot (แชท)** ก่อน — เป็นช่องทางที่กฎ Meta เอื้อที่สุด, รูปสลิปเก็บได้, ตรงกับพฤติกรรม chat commerce ไทย แล้วค่อยขยายไปตอบใต้คอมเมนต์เป็นเฟส 2 · ต่อ **บน EasySlip/Thunder API ที่มีอยู่** (ไม่ต้องสร้าง engine ใหม่)

_(รายละเอียดแนวทาง + ข้อดีข้อเสีย ดูข้อ 7)_

## 2. ความเป็นไปได้ + ช่องทาง Facebook ที่ใช้ได้จริง [หามาเพิ่ม]

Facebook มี 4 ช่องทางที่รับรูปสลิปได้ ไล่จากดีสุด → ยากสุด:

### 🟢 ช่องทาง A — Messenger Bot (แชทอินบ็อกซ์) ← แนะนำที่สุด
- ลูกค้าส่งรูปสลิปเข้าแชทเพจ → เพจได้ webhook event `messages` ที่มี attachment รูป
- payload รูปมาเป็น `{ "type":"image", "payload":{ "url":"..." } }`
- **Meta บอกเองให้ "capture and store any webhook payload content that you want to keep"** → ฝั่ง Messenger เก็บรูป/ประมวลผลได้ ไม่ติดข้อห้าม
- สิทธิ์ที่ต้องขอ: `pages_messaging` + `pages_manage_metadata`
- ✅ เหมาะสุด เพราะควบคุมได้เต็ม, รูปเข้าถึงได้, ตรงกับ use case "transactional confirmations"

### 🟡 ช่องทาง B — ตอบใต้คอมเมนต์โพสต์ (Comment → Private Reply)
- ลูกค้าคอมเมนต์รูปสลิปใต้โพสต์ → subscribe webhook field **`feed`** (ไม่มี field `comments` แยก — คอมเมนต์มาในรูป `feed` item=`comment`)
- ⚠️ **จุดที่ต้องรู้ (จากงานวิจัย verify แล้ว):** รูปในคอมเมนต์ **webhook มักไม่ส่ง URL รูปมาตรงๆ** — ต้องยิง Graph API เพิ่ม 1 ครั้ง: `GET /{comment-id}?fields=attachment` → `data.attachment.media.image.src` (ต่างจากรูปที่โพสต์เป็น post ซึ่งได้ลิงก์มาเลย)
- ⚠️ **CDN อาจย่อขนาดรูป (downscale)** → อาจกระทบความแม่นในการอ่าน QR/OCR ของสลิป — ต้องเทสต์จริง
- ⚠️ URL รูป **หมดอายุ** (signed CDN link) — ต้องดึง/ยิงเข้า EasySlip ทันที
- **ตอบกลับ:** คอมเมนต์ตอบกลับ หรือ **Private Reply** (เด้งเข้าแชท) — แต่ Private Reply **ส่งได้แค่ 1 ข้อความ ภายใน 7 วัน และเป็น text ล้วน (แนบรูปไม่ได้)** พอสำหรับแจ้งผล แต่จะคุยต่อได้ก็ต่อเมื่อลูกค้าตอบกลับ (เปิดกรอบ 24 ชม.)
- สิทธิ์: `pages_manage_metadata` + `pages_messaging` (+ อาจ `pages_manage_engagement`/`pages_read_user_content`)

### 🟡 ช่องทาง C — Facebook Live (คอมเมนต์สด)
- ต่อยอดจากช่องทาง B — ช่วงไลฟ์ขายของ ลูกค้าโอนแล้วส่งสลิปใต้ไลฟ์
- ทำได้แต่ volume พุ่งสูงช่วงไลฟ์ ต้องระวัง rate limit + ลิงก์รูปหมดอายุเร็ว

### 🔴 ช่องทาง D — Instagram Messaging (แถม)
- Instagram วิ่งบน Messenger Platform เดียวกัน — ลูกค้าส่งรูปสลิปใน DM ได้ (png/jpeg สูงสุด 8MB) มาเป็น `{type:"image", payload:{url}}` เหมือนกัน กรอบ 24 ชม. เดียวกัน
- Private Reply ของ IG: ตอบคอมเมนต์แบบส่วนตัวได้ **1 ข้อความ ภายใน 7 วัน** (โพสต์/reel/story/ad; ส่วนไลฟ์ตอบได้เฉพาะช่วงกำลังไลฟ์)
- ⚠️ **รูปแบบ "view-once/ephemeral" ของ IG จะมาโดย "ไม่มี url"** → ต้องบอกลูกค้าส่งรูปสลิปแบบปกติเท่านั้น
- สิทธิ์เพิ่ม: `instagram_basic` + `instagram_manage_messages` (หรือชุด `instagram_business_*` ยุค Instagram Login)

### สิทธิ์ (permissions) ที่ต้องขอ — ชุดจริงที่ deployment ใช้ [verify แล้ว]
- **Messenger:** `pages_show_list`, `pages_manage_metadata` (subscribe/รับ webhook), `pages_messaging` (รับ-ส่งข้อความ), `pages_read_engagement`, `business_management` (เป็น dependency)
- **Instagram:** เพิ่ม `instagram_basic` + `instagram_manage_messages`
- ชุดขั้นต่ำแค่รับ-ส่งได้ = `pages_manage_metadata` + `pages_read_engagement` + `pages_messaging`

### rate limit [หามาเพิ่ม — ยังไม่ได้ cross-verify รอบสอง]
| API | เพดาน (ต่อ 24 ชม.) |
|-----|-----|
| Messenger | 200 × จำนวน engaged users |
| Pages (page token) | 4,800 × จำนวน engaged users |
| Instagram private replies (live) | 100 ครั้ง/วินาที |

_⚠️ ตัวเลข rate limit นี้มาจากเอกสาร Meta แต่ทีมวิจัยยังไม่ได้ยืนยันซ้ำ — ควรเช็คใหม่ตอนออกแบบจริง · webhook ต้องตอบ HTTP 200 ภายใน 5 วินาที ไม่งั้นถูกตัด subscription_

## 3. นโยบาย/ข้อจำกัดของ Meta ที่กระทบระบบตรวจสลิป [หามาเพิ่ม]

### ✅ ทำได้
- **ยืนยันธุรกรรม/การชำระเงินได้** — `pages_messaging` ระบุ use case "transactional confirmations" ตรงตัว
- **เก็บ webhook payload ฝั่ง Messenger ได้** — Meta แนะนำให้เก็บเอง

### ⚠️ ต้องระวัง / มีเงื่อนไข

**(1) ต้องผ่าน App Review + Business Verification** [verify แล้ว]
- ทำ/เทสต์บนเพจของ **เราเอง** (เฉพาะ admin/dev/tester) → **ไม่ต้อง** App Review
- แต่พอรับลูกค้าจริง โดยเฉพาะแบบ **multi-tenant (หลายเพจของหลายร้าน)** → ต้องขอ **Advanced Access** ซึ่ง **บังคับทั้ง App Review และ Business Verification** + อาจต้องทำ Data Protection Assessment
- 📌 นี่คือ "ประตูด่านแรก" ที่กินเวลา — ควรเริ่มยื่นเนิ่นๆ (pilot ร้านเดียวบนเพจตัวเองเริ่มได้ก่อนโดยไม่ต้องรอ review)

**(2) message tag เดิมตายแล้ว — ต้องใช้ Utility Messages**
- tag `POST_PURCHASE_UPDATE`, `CONFIRMED_EVENT_UPDATE`, `ACCOUNT_UPDATE` **ถูกยกเลิกตั้งแต่ 27 เม.ย. 2026** ยิง API แล้ว error 100 ทันที
- ✅ ตัวแทนคือ **Utility Messages / Utility Templates** — และ **เปิดใช้ในไทยแล้ว** (มีแค่ 4 ประเทศ: ไทย ฟิลิปปินส์ สหรัฐฯ เวียดนาม)
- ใช้ส่งข้อความ *"สลิปผ่านแล้ว ยอด 1,500 บาท ออเดอร์ #123"* หาลูกค้า **นอกกรอบ 24 ชม.** ได้อย่างถูกกฎ
- เงื่อนไข: ต้องมีสิทธิ์ `page_utility_messaging`, template ต้องอนุมัติก่อน (เพจสร้างเองอนุมัติในไม่กี่วินาที), **ห้ามมีเนื้อหาการตลาดปนในข้อความ utility**

**(3) กรอบ 24 ชั่วโมง (Standard Messaging Window)**
- ตอบลูกค้าฟรีได้ภายใน 24 ชม. หลังลูกค้าทัก — เกินนั้นต้องใช้ Utility Message หรือ Human Agent tag (7 วัน)
- งานตรวจสลิปส่วนใหญ่เกิดในไม่กี่นาที → อยู่ในกรอบ 24 ชม. อยู่แล้ว ไม่ค่อยเป็นปัญหา

**(4) Meta Platform Terms — ผูกมัดวิธีจัดการข้อมูลสลิป** [verify แล้ว จากสัญญา Meta อัปเดต ก.พ. 2026]
- **ต้องลบ Platform Data (รวมรูปสลิป) "ทันทีที่ทำได้" เมื่อไม่จำเป็นต่อธุรกิจแล้ว** และลบ/อัปเดตทันทีเมื่อ Meta หรือผู้ใช้ร้องขอ — ห้ามเก็บถาวรโดยปริยาย (Sec 3.d)
- ในฐานะ **"Tech Provider"** ที่ให้บริการร้านค้า: ประมวลผลข้อมูลได้ **"ในนามและตามคำสั่งของร้านค้าเท่านั้น ห้ามใช้เพื่อวัตถุประสงค์ของตัวเอง"** + แยกข้อมูลรายร้าน (Sec 5.b) → ห้ามเอาข้อมูลสลิปจาก Facebook ไปใช้ประโยชน์อื่น
- 💡 นัย: โมเดลธุรกิจต้องวางเป็น "ผู้ประมวลผลแทนร้านค้า" ชัดเจน — ตรงกับ positioning B2B ของ Thunder อยู่แล้ว

**(5) PDPA ไทย + ข้อมูลการเงิน**
- ข้อมูลสลิป = ข้อมูลส่วนบุคคล/การเงิน → เข้าข่าย **PDPA ไทย** ต้องมีฐานการประมวลผล + นโยบายความเป็นส่วนตัว + เก็บเท่าที่จำเป็น
- 🔒 สอดคล้องกับกฎห้องเราอยู่แล้ว: ข้อมูลสลิป/การเงินเป็นความลับ
- _⚠️ PDPA + Meta Payments/commerce policy ยังไม่ได้ verify เชิงลึกในรอบนี้ — ควรตรวจกับฝ่ายกฎหมายก่อนขึ้นจริง_

## 4. ต่อยอดจาก EasySlip อย่างไร [หามาเพิ่ม — จาก document.easyslip.com]

**ข่าวดี: EasySlip API v2 แทบจะออกแบบมาให้ต่อ Facebook พอดี**

### สถาปัตยกรรมที่เสนอ (Messenger)
```
ลูกค้าส่งสลิปในแชทเพจ
   → Facebook ยิง webhook มาที่ server เรา (มี URL รูป)
   → server เอา URL ยิงเข้า EasySlip: POST /v2/verify/bank { url, checkDuplicate:true, matchAmount, matchAccount }
   → EasySlip คืนผล (ยอด/บัญชี/ซ้ำไหม/ตรงยอดไหม)
   → บอทตอบลูกค้าในแชท + อัปเดตสถานะออเดอร์
```

### จุดที่ทำให้ต่อง่าย
- **`POST /v2/verify/bank` รับ input ได้ 3 แบบ:** `payload` (สตริง QR), `image` (อัปโหลดไฟล์), และ **`url`** (ส่ง URL รูป ≤4MB ให้มันไปดึงเอง)
- ฝั่ง Messenger เอา URL จาก webhook ยิงเข้า `url` ได้ตรงๆ **ไม่ต้องโหลดรูปมาเก็บเอง** (ลดภาระ + ลดประเด็น PDPA)
- ฝั่ง Instagram/คอมเมนต์ที่ลิงก์หมดอายุ → ใช้ `image` แบบ pass-through (โหลด-ยิง-ทิ้ง ไม่เก็บ)

### ฟีเจอร์ที่ตรงกับความต้องการร้านค้า
| option | ทำอะไร |
|--------|--------|
| `checkDuplicate` | กันสลิปซ้ำ (สลิปเก่าเอามาใช้ใหม่) |
| `matchAmount` | เช็คยอดโอนตรงกับออเดอร์ไหม |
| `matchAccount` | เช็คว่าโอนเข้าบัญชีร้านจริง |
| output `isDuplicate/rawSlip` | คืน transRef, ยอด, ชื่อผู้ส่ง/รับ, ธนาคาร |

### ข้อจำกัดที่ต้องรู้ [หามาเพิ่ม]
- รองรับ 18+ ธนาคาร; **อายุสลิปที่ตรวจได้จำกัด** — KBANK/PromptPay 180 วัน, SCB/BBL/กรุงศรี 90 วัน, TrueMoney 30 วัน
- ต้อง whitelist IP ของ server + เก็บ API key ฝั่ง server เท่านั้น (ห้าม client-side)
- เบื้องหลัง EasySlip ตรวจผ่าน **ITMX** (ระบบกลางธนาคาร) *[จากไฟล์บริบท ยืนยันด้วยเว็บ ITMX]* → ถ้า ITMX/ธนาคารล่ม การตรวจก็กระทบ (เป็นเคส CS ที่ Thunder เจอบ่อยตามไฟล์บริบท)

### กลุ่ม A — แพลตฟอร์มแชทที่ตรวจสลิปบน Facebook **ได้แล้ว** (คู่แข่งตรงของ pivot นี้)
| เจ้า | ทำอะไรบน FB | ราคา/จุดเด่น | จุดอ่อน |
