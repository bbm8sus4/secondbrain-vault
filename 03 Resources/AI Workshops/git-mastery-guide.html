Chapter 01

Git Object Model — ทุกอย่างเริ่มจากตรงนี้

คนส่วนใหญ่ใช้ Git แบบจำคำสั่ง ไม่เข้าใจว่าข้างในเก็บข้อมูลยังไง พอเจอปัญหาก็แก้ไม่ได้ เข้าใจ object model แล้วทุกอย่างจะ click

4 Object Types

Git เก็บทุกอย่างเป็น object ใน .git/objects/ แต่ละ object มี SHA-1 hash เป็น address

  commit ──references──▶ tree ──references──▶ blob
    │                      │
    │ parent               │ subtree
    ▼                      ▼
  commit               tree ──▶ blob

  + tag (optional: annotated tag pointing to a commit)
Objectเก็บอะไรKey Insight
blobเนื้อหาไฟล์ (ไม่มีชื่อไฟล์)ไฟล์ที่เนื้อหาเหมือนกัน = blob เดียวกัน ไม่ว่าชื่อจะต่างกัน
treedirectory listing (ชื่อ + mode + pointer ไป blob/tree)ทำให้ Git track directory structure ได้
committree pointer + parent(s) + author + messagecommit ไม่ได้เก็บ diff เก็บ snapshot ทั้งก้อน
tagpointer ไป commit + tagger + messageannotated tag เท่านั้นที่เป็น object (lightweight tag เป็นแค่ ref)

ลองดูของจริง

# ดู object ทุกตัวใน repo
git rev-list --all --objects

# ดูประเภทของ object
git cat-file -t HEAD
commit

# ดูเนื้อหาของ commit object
git cat-file -p HEAD
tree 4b825dc642cb6eb9a060e54bf899d8e5f2..
parent a1b2c3d4e5f6...
author John <john@x.com> 1720000000 +0700
committer John <john@x.com> 1720000000 +0700

Fix payment validation

# ดูเนื้อหาของ tree
git cat-file -p HEAD^{tree}
100644 blob a1b2c3...  README.md
040000 tree d4e5f6...  src

# ดูเนื้อหาของ blob
git cat-file -p a1b2c3
(contents of README.md)
Teaching Point

Git ไม่ได้เก็บ diff ระหว่าง version อย่างที่หลายคนเข้าใจ ทุก commit เก็บ snapshot ของไฟล์ทั้งหมด แต่ถ้าไฟล์ไม่เปลี่ยน มันก็ point ไปที่ blob เดิม ไม่ copy ซ้ำ (content-addressable storage)

Refs — ชื่อเรียก commit

Branch, tag, HEAD ทั้งหมดเป็นแค่ text file ที่เก็บ SHA

cat .git/refs/heads/main
a1b2c3d4e5f6789...     # main = pointer ไปที่ commit นี้

cat .git/HEAD
ref: refs/heads/main  # HEAD = pointer ไปที่ branch ไหน
Key Insight

Branch ไม่ใช่ "สำเนาของโค้ด" แต่เป็นแค่ pointer ไปที่ commit ตัวเดียว สร้าง branch = สร้างไฟล์ 41 bytes ไม่ copy อะไรเลย นี่คือเหตุผลที่ Git branching ถูกมาก

Chapter 02

Three Trees — สามที่ที่โค้ดอยู่

ทุกปัญหาเรื่อง staging, reset, checkout จะเคลียร์ถ้าเข้าใจว่า Git มี "tree" สามตัวที่โค้ดวิ่งไปมา

  ┌─────────────────┐    git add    ┌─────────────────┐   git commit   ┌─────────────────┐
  │  Working Dir     │ ───────────▶ │  Staging Area    │ ────────────▶ │  HEAD (commit)   │
  │  (ไฟล์ที่เห็นจริง)  │              │  (Index/.git/index)│               │  (snapshot ล่าสุด) │
  └─────────────────┘              └─────────────────┘               └─────────────────┘
         ▲                                                                     │
         └──────────────── git checkout / git restore ─────────────────────────┘

ดูสถานะทั้ง 3 ที่พร้อมกัน

# ดูว่า staging กับ HEAD ต่างกันยังไง (สิ่งที่จะ commit)
git diff --cached

# ดูว่า working dir กับ staging ต่างกันยังไง (สิ่งที่ยังไม่ add)
git diff

# ดูว่า working dir กับ HEAD ต่างกันยังไง (ทุกอย่างที่เปลี่ยน)
git diff HEAD

# ดูว่าแต่ละไฟล์ อยู่ใน tree ไหนบ้าง
git status -s
 M file.js   # M ด้านขวา = modified ใน working dir (ยังไม่ add)
M  file.js   # M ด้านซ้าย = modified ใน staging (add แล้ว)
MM file.js   # ทั้งสองที่ = add แล้วบางส่วน แก้ต่อหลัง add

Partial Staging — add เฉพาะบางบรรทัด

# เลือก add ทีละ hunk (ส่วน) ของไฟล์
git add -p file.js
Stage this hunk [y,n,s,e,q,?]?

# y = stage hunk นี้, n = ข้าม, s = split เล็กลง, e = edit hunk เอง
Workshop Tip

git add -p เป็น skill ที่แยกมือใหม่กับมือโปรชัดเจน ทำให้ commit สะอาด แต่ละ commit เป็นเรื่องเดียว ไม่ใช่ "แก้ทุกอย่างแล้ว add . ยิง commit ทีเดียว"

Chapter 03

Reflog — ตาข่ายนิรภัย

Reflog บันทึกทุกการเคลื่อนไหวของ HEAD และ branch ในเครื่องเรา เหมือน undo history ของ Git เอง เก็บ 90 วันโดยค่าเริ่มต้น

git reflog
a1b2c3d HEAD@{0}: commit: Fix payment validation
f4e5d6c HEAD@{1}: rebase (finish): returning to refs/heads/main
8a9b0c1 HEAD@{2}: rebase (pick): Add user auth
d2e3f4a HEAD@{3}: reset: moving to HEAD~3
7b8c9d0 HEAD@{4}: commit: Add dashboard charts

# reflog ของ branch เฉพาะ
git reflog show feature/payment

# reflog พร้อมเวลา
git reflog --date=relative

กู้ commit ที่ "หายไป"

ใน Git ไม่มีอะไรหายจริง (จนกว่า garbage collection จะลบ ซึ่งปกติ 90 วัน)

# เผลอ reset --hard ไปแล้ว อยากกลับมา
git reflog
# หา SHA ของ commit ที่อยากกลับไป
git reset --hard HEAD@{4}

# ลบ branch ไปแล้ว อยากกู้
git reflog
# หา SHA ที่ branch เคยชี้อยู่
git branch recovered-branch HEAD@{7}
Rule of Thumb

เผลอทำอะไรพัง ให้หยุดหายใจลึก แล้ว git reflog ก่อน 99% ของกรณีกู้ได้ ห้ามรีบ git clone ใหม่

Chapter 04

Reset Demystified — สาม mode ที่ต้องแยกให้ออก

git reset คือคำสั่งที่คนกลัวมากที่สุด เพราะไม่เข้าใจว่ามันมีสาม mode ที่ทำต่างกันมาก

ModeHEADStagingWorking Dirใช้เมื่อ
--soft เลื่อน ไม่แตะ ไม่แตะ อยากรวม commit หลายตัวเป็นตัวเดียว (squash แบบง่าย)
--mixed (default) เลื่อน reset ไม่แตะ ถอน staging กลับมาเป็น unstaged (undo git add)
--hard เลื่อน reset reset ทิ้งทุกอย่างกลับไปที่ commit นั้น 
# Squash 3 commits สุดท้ายเป็นตัวเดียว (soft reset → commit ใหม่)
git reset --soft HEAD~3
git commit -m "Feature: complete payment flow"

# ถอน staging (undo git add) ไม่เสียงาน
git reset HEAD file.js
# หรือแบบใหม่ที่ชัดกว่า:
git restore --staged file.js

# ทิ้งทุกอย่าง กลับไปที่ commit ล่าสุด
git reset --hard HEAD
Danger Zone

--hard เป็น mode เดียวที่ทำให้งานหาย uncommitted changes ที่ถูก reset --hard ไป reflog ช่วยไม่ได้ (เพราะมันไม่เคยเป็น commit) ให้ git stash ก่อนเสมอถ้าไม่แน่ใจ

Chapter 05

Interactive Rebase — แก้ไขประวัติศาสตร์

Interactive rebase เป็นเครื่องมือที่ทรงพลังที่สุดของ Git ใช้จัดเรียง, รวม, แก้, ลบ commit ก่อน push

# จัดการ 5 commits สุดท้าย
git rebase -i HEAD~5

# editor จะเปิดมาแบบนี้:
pick a1b2c3d Add login page
pick d4e5f6a Fix typo in login
pick 7b8c9d0 Add payment form
pick e1f2a3b WIP: debugging payment
pick c4d5e6f Complete payment validation

# เปลี่ยน command หน้า commit ที่ต้องการ:
CommandShortทำอะไร
pickpเก็บ commit นี้ไว้ (ค่าเริ่มต้น)
rewordrเก็บ commit แต่แก้ message
editeหยุดที่ commit นี้ให้แก้ไข (amend ได้)
squashsรวมเข้ากับ commit ก่อนหน้า (รวม message ด้วย)
fixupfรวมเข้า commit ก่อนหน้า (ทิ้ง message)
dropdลบ commit นี้ทิ้ง

ตัวอย่าง: Clean up ก่อน push

# เปลี่ยนจาก:
pick a1b2c3d Add login page
pick d4e5f6a Fix typo in login          ← รวมเข้า login
pick 7b8c9d0 Add payment form
pick e1f2a3b WIP: debugging payment     ← ลบ
pick c4d5e6f Complete payment validation ← รวมเข้า payment

# เป็น:
pick a1b2c3d Add login page
fixup d4e5f6a Fix typo in login
pick 7b8c9d0 Add payment form
drop e1f2a3b WIP: debugging payment
fixup c4d5e6f Complete payment validation

# ผลลัพธ์: 2 commits สะอาด แทนที่ 5 commits รก

Autosquash — ทำ fixup อัตโนมัติ

# commit ที่ตั้งใจจะ fixup เข้า commit เก่า
git commit --fixup=a1b2c3d

# แล้วพอ rebase มันจะเรียงให้อัตโนมัติ
git rebase -i --autosquash HEAD~5

# ตั้งค่าให้ autosquash เป็นค่าเริ่มต้น
git config --global rebase.autosquash true
Golden Rule

ห้าม rebase commit ที่ push ไปแล้วและมีคนอื่นใช้อยู่ Rebase เปลี่ยน SHA ทุกตัว = history ของคนอื่นจะพัง ใช้ได้เฉพาะกับ local commits หรือ branch ส่วนตัว

Chapter 06

Cherry-pick & Revert — ศัลยกรรม commit

Cherry-pick: หยิบ commit จาก branch อื่น

# หยิบ commit เดียว
git cherry-pick abc1234

# หยิบหลาย commits (ทำ commit ใหม่ทีละตัว)
git cherry-pick abc1234 def5678 ghi9012

# หยิบ range (ไม่รวมตัวแรก, รวมตัวสุดท้าย)
git cherry-pick abc1234..ghi9012

# หยิบมาแต่ไม่ commit (เอามาวางใน staging)
git cherry-pick --no-commit abc1234

Revert: ย้อน commit อย่างปลอดภัย

ต่างจาก reset ตรงที่ revert สร้าง commit ใหม่ที่เป็น "ตรงข้าม" ปลอดภัยบน shared branch

# ย้อน commit เดียว
git revert abc1234

# ย้อนหลาย commits (ย้อนทีละตัว, commit ล่าสุดก่อน)
git revert abc1234..def5678

# ย้อน merge commit (ต้องบอกว่าเอา parent ฝั่งไหน)
git revert -m 1 abc1234
# -m 1 = เก็บฝั่ง mainline (parent แรก), ย้อนฝั่ง feature
เมื่อไหร่ใช้อะไร

Cherry-pick = ต้องการบาง commit จาก branch อื่น (hotfix ที่ต้องไปหลาย branch)
Revert = ต้องการถอน commit บน shared branch (ปลอดภัย ไม่ rewrite history)
Reset = ต้องการลบ commit จาก local branch ของตัวเอง (rewrite history)

Chapter 07

Bisect — หา commit ที่ทำพัง ด้วย Binary Search

มีบั๊กแต่ไม่รู้ว่า commit ไหนทำพัง bisect ใช้ binary search หาให้ใน O(log n) — 1000 commits ใช้แค่ ~10 steps

# เริ่ม bisect
git bisect start

# บอกว่า commit ปัจจุบันพัง
git bisect bad

# บอกว่า commit เก่าที่ยังดีอยู่
git bisect good v2.0.0
Bisecting: 512 revisions left to test after this (roughly 9 steps)

# Git จะ checkout commit กลาง — ทดสอบ แล้วบอก
git bisect good  # ถ้า commit นี้ยังดี
git bisect bad   # ถ้า commit นี้พัง

# ทำซ้ำจนเจอ → Git จะบอกว่า commit ไหนเป็นตัวแรกที่พัง
abc1234 is the first bad commit
Author: John <john@x.com>
Date: ...
    Refactor payment handler

# เสร็จแล้ว กลับมาที่เดิม
git bisect reset

Bisect อัตโนมัติด้วย script

# ให้ script ทดสอบแทน (exit 0 = good, exit 1 = bad)
git bisect start HEAD v2.0.0
git bisect run npm test
# หรือ custom script:
git bisect run ./test-payment.sh

# Git จะทำ binary search + run script ให้จนเจอ commit ที่ผิด
Pro Move

Bisect + automated test = superpower ตัวจริง เขียน test ที่ reproduce บั๊กก่อน แล้วให้ bisect run หา commit ที่ทำพังให้อัตโนมัติ เหมาะมากตอนสอน "ทำไม test สำคัญ"

Chapter 08

Stash ขั้นสูง

คนส่วนใหญ่ใช้แค่ git stash กับ git stash pop แต่ stash ทำได้อีกเยอะ

# stash พร้อมข้อความ (หาง่ายกว่า)
git stash push -m "WIP: payment refactor"

# stash เฉพาะบางไฟล์
git stash push -m "config changes" -- config.js .env

# stash เฉพาะบาง hunk (เหมือน add -p)
git stash push -p

# stash รวม untracked files ด้วย
git stash push -u -m "including new files"

# ดูรายการ stash ทั้งหมด
git stash list
stash@{0}: On main: WIP: payment refactor
stash@{1}: On feature: config changes

# ดูว่า stash นี้เปลี่ยนอะไร
git stash show -p stash@{1}

# apply โดยไม่ลบออกจาก stash list (ปลอดภัยกว่า pop)
git stash apply stash@{1}

# สร้าง branch จาก stash (เอา stash ไปทำต่อใน branch ใหม่)
git stash branch new-feature stash@{0}
Tip

ใช้ apply แทน pop เป็นค่าเริ่มต้น ถ้า apply แล้ว conflict หรือมีปัญหา stash ยังอยู่ ค่อย git stash drop ทีหลัง

Chapter 09

Log Forensics — สืบค้นประวัติ

Git log ไม่ใช่แค่ดู commit ล่าสุด มันเป็นเครื่องมือสืบสวนที่ทรงพลังมาก

git log ขั้นเทพ

# Log สวย มีกราฟ ดู branching
git log --oneline --graph --all --decorate

# หา commit ที่แก้ไฟล์นี้
git log --oneline -- src/payment.js

# หา commit ที่มีคำนี้ใน message
git log --oneline --grep="payment"

# หา commit ที่มีคำนี้ใน code (pickaxe search)
git log -S "calculateTotal" --oneline

# หา commit ที่มี regex นี้ใน code change
git log -G "TODO|FIXME|HACK" --oneline

# หา commit ของคนนี้ ในช่วงเวลานี้
git log --author="John" --since="2024-01-01" --until="2024-06-30" --oneline

# แสดง diff ด้วย (ดูว่าแต่ละ commit เปลี่ยนอะไร)
git log -p -- src/payment.js

# แสดงแค่สถิติ (กี่ไฟล์ กี่บรรทัด)
git log --stat --oneline

git blame — ใครแก้บรรทัดนี้

# ดูว่าแต่ละบรรทัดใครแก้ เมื่อไหร่
git blame src/payment.js

# ดูเฉพาะบรรทัดที่ 50-70
git blame -L 50,70 src/payment.js

# ข้ามพวก whitespace change / bulk rename
git blame -w -M src/payment.js

# ข้ามพวก code move ข้ามไฟล์
git blame -w -C -C src/payment.js

git shortlog — สรุปผลงาน

# สรุปจำนวน commit ของแต่ละคน
git shortlog -sn --all
   847  John
   623  Jane
   201  Bob

# เฉพาะช่วงเวลา
git shortlog -sn --since="2024-01-01"
Power Combo

git log -S "functionName" (pickaxe) คือเครื่องมือที่ถูก undervalue มากที่สุด หาได้ว่า function นี้ถูกเพิ่ม/ลบใน commit ไหน เหมาะกับคำถามว่า "ทำไมถึงลบ function นี้ออก?"

Chapter 10

Recovery Recipes — กู้ทุกสถานการณ์

รวม recipe สำหรับสถานการณ์ที่พบบ่อย เรียงจากง่ายไปยาก

ย้อน commit ล่าสุด (ยังไม่ push) 
# เก็บ changes ไว้ใน staging
git reset --soft HEAD~1

# เก็บ changes ไว้ใน working dir
git reset HEAD~1

# ทิ้ง changes เลย
git reset --hard HEAD~1
ย้อน commit ที่ push ไปแล้ว 
# สร้าง commit ตรงข้าม (safe for shared branches)
git revert abc1234
git push
เผลอ commit ไปผิด branch 
# อยู่ main แต่เผลอ commit ที่ควรไป feature

# 1) สร้าง branch ใหม่ที่ commit ปัจจุบัน (เก็บงานไว้)
git branch feature/my-work

# 2) ย้อน main กลับไปก่อน commit
git reset --hard HEAD~1

# 3) ไปทำงานต่อที่ branch ใหม่
git checkout feature/my-work
เผลอ git reset --hard ไป 
# reflog จะช่วยได้ (ถ้าเคย commit ไว้)
git reflog
# หา SHA ที่ต้องการ
git reset --hard HEAD@{2}

# ถ้ายังไม่เคย commit → ไม่มีทาง reflog ช่วยไม่ได้
# บทเรียน: commit บ่อยๆ, stash ก่อน reset
ลบ branch ไปแล้ว อยากกู้ 
git reflog
# หา SHA ที่ branch เคยชี้
git branch recovered abc1234
Rebase conflict เยอะ อยากยกเลิก 
git rebase --abort
# กลับไปสถานะก่อน rebase ทันที ไม่เสียอะไร
Merge conflict เยอะ อยากยกเลิก 
git merge --abort
# กลับไปสถานะก่อน merge
เผลอ push secret/credential ขึ้นไป
Critical

ถ้า push ไป public repo แล้ว ให้ถือว่า secret โดน compromise → revoke + rotate ทันที การลบ commit ไม่พอ (GitHub/caches อาจเก็บไว้แล้ว)

# ลบไฟล์ออกจากทุก commit ใน history
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch secrets.env" \
  --prune-empty --tag-name-filter cat -- --all

# หรือใช้ BFG Repo-Cleaner (เร็วกว่ามาก)
bfg --delete-files secrets.env
git reflog expire --expire=now --all
git gc --prune=now --aggressive
กู้ไฟล์ที่ถูกลบใน commit เก่า

git log --diff-filter=D --summary -- path/to/file.js

Hooks & Automation

Git hooks คือ script ที่รันอัตโนมัติเมื่อเกิด event ใน Git อยู่ใน .git/hooks/ หรือ share ผ่าน .githooks/

Hooks ที่ใช้บ่อย

Hookเมื่อไหร่ใช้ทำอะไร
pre-commitก่อน commitLint, format, ตรวจ secrets
commit-msgหลังเขียน messageตรวจ format ของ commit message
pre-pushก่อน pushรัน test, ตรวจ branch protection
prepare-commit-msgก่อนเปิด editorใส่ template / ticket number อัตโนมัติ
post-mergeหลัง merge/pullรัน npm install ถ้า package.json เปลี่ยน

ตัวอย่าง: ตรวจ secrets ก่อน commit

# .git/hooks/pre-commit (ต้อง chmod +x)
#!/bin/sh

# หา pattern ที่คล้าย secret ในไฟล์ที่กำลังจะ commit
if git diff --cached --diff-filter=ACM | \
   grep -iE '(password|secret|api_key|token)\s*[:=]' ; then
  echo "Possible secret detected. Commit blocked."
  exit 1
fi

Chapter 12

Worktrees & Performance

Worktrees — ทำงานหลาย branch พร้อมกัน

ปกติ checkout branch ใหม่ต้องเก็บงานก่อน (stash/commit) Worktree ให้ทำงานหลาย branch พร้อมกันโดยแต่ละ branch อยู่คนละ folder

# สร้าง worktree สำหรับ branch hotfix (folder ใหม่)
git worktree add ../hotfix-payment hotfix/payment

# ทำงานใน folder นั้นเลย ไม่กระทบ working dir หลัก
cd ../hotfix-payment
# แก้ไข, commit, push ได้อิสระ

# ดูรายการ worktree ทั้งหมด
git worktree list

# ลบ worktree เมื่อเสร็จ
git worktree remove ../hotfix-payment

Performance สำหรับ Large Repos

# Sparse checkout — clone เฉพาะ folder ที่ต้องการ
git clone --filter=blob:none --sparse https://github.com/big/repo.git
cd repo
git sparse-checkout set src/frontend
# ได้เฉพาะ src/frontend, folder อื่นจะ download เมื่อเข้าถึง

# Shallow clone — เอาแค่ history ล่าสุด
git clone --depth=1 https://github.com/big/repo.git
# ต้องการ history เพิ่มทีหลัง:
git fetch --deepen=50

# Maintenance — optimize repo ใหญ่
git maintenance start
# จัด background tasks: gc, commit-graph, prefetch

# File System Monitor — เร็วขึ้นสำหรับ repo ใหญ่
git config core.fsmonitor true
git config core.untrackedcache true

Partial Clone & Bundle

# Bundle — สร้างไฟล์ส่ง Git repo ให้คนอื่นโดยไม่ต้องมี server
git bundle create repo.bundle --all
# อีกเครื่อง:
git clone repo.bundle

# Bundle เฉพาะ commits ใหม่
git bundle create update.bundle main~10..main
