## 1. วัตถุประสงค์

- ทีม CS ออกประกาศได้เองจากเทมเพลต ไม่ต้องร่างใหม่และรออนุมัติข้อความทีละคำทุกครั้ง
- ทุกประกาศมีมาตรฐานเดียวกัน: โทนเดียวกัน ช่องทางถูก จังหวะถูก มี audit trail
- ลดขั้นตอน: จากเดิม "ร่างเอง → ส่งให้ C แก้ไปมา → ส่ง" เหลือ "เติมเทมเพลต → C อนุมัติรอบเดียว → ส่ง → ลง log"

## 2. บทบาทหน้าที่ (Roles)

| บทบาท | ผู้รับผิดชอบ | หน้าที่ |
|---|---|---|
| **ผู้อนุมัติ (Approver)** | C (COO) | อนุมัติเนื้อหา + จังหวะเวลา + กลุ่มเป้าหมาย ก่อนส่งทุกครั้ง |
| **ผู้ร่าง/ผู้ส่ง (Executor)** | Ning (หลัก) / ทีม CS | เติมเทมเพลต, เช็ค checklist, ส่งตามช่องทาง, ลง log, ติดตามผล |
| **ผู้แจ้งเหตุ (Reporter)** | Prae / Arunjai / ทีม Dev | แจ้งเหตุขัดข้องหรือกำหนดการ deploy เข้ากลุ่มทีมทันทีที่รู้ |
| **กราฟิก (Artwork)** | Graphic Designer | ทำภาพประกอบตาม artwork brief (เฉพาะประกาศที่ต้องมีภาพ) |
| **ผู้สำรอง (Backup)** | [เติมชื่อ] | ทำหน้าที่แทน Ning เมื่อลา/ไม่อยู่ |

> หลักการ: **คนร่างกับคนอนุมัติต้องไม่ใช่คนเดียวกัน** — ประกาศที่ออกในนามบริษัทมีผลต่อความเชื่อมั่นลูกค้าเสมอ

## 3. ระดับความเร่งด่วน (Tier) — ตัวกำหนดทุกอย่าง

| Tier | ประเภท | ขออนุมัติล่วงหน้า | ช่องทางขออนุมัติ | เทมเพลต |
|---|---|---|---|---|
| **T1 ด่วน** | เหตุขัดข้อง ระบบล่ม/ช้า/ตรวจสลิปไม่ได้ | ทันที | LINE ถึง C โดยตรง + โทรถ้าไม่ตอบใน 15 นาที | [[11-เทมเพลต-เหตุขัดข้อง]] |
| **T2 ตามแผน** | ปิดปรับปรุง / deploy / ฟีเจอร์ใหม่ / เปลี่ยนราคา | ≥ 3 วันทำการ | กลุ่มทีมงาน (แนบร่างเต็ม) | [[10-เทมเพลต-ปิดปรับปรุง]] / [[12-เทมเพลต-ฟีเจอร์ใหม่]] |
| **T3 การตลาด** | โปรโมชั่น แคมเปญ ข่าวสาร | ≥ 5 วันทำการ (เผื่อทำ artwork) | กลุ่มทีมงาน (แนบร่าง + artwork brief) | [[13-เทมเพลต-โปรโมชั่น]] |

### กติกาพิเศษ T1 (fast-track)

เหตุขัดข้องกำลังเกิด ลูกค้ากำลังได้รับผลกระทบ — ความเร็วสำคัญกว่าความสมบูรณ์:

1. ส่งข้อความ **"รับทราบปัญหา" (จังหวะที่ 1 ใน [[11-เทมเพลต-เหตุขัดข้อง]]) ได้ทันทีโดยไม่ต้องรออนุมัติ** — ข้อความนี้ C อนุมัติเนื้อหาไว้ล่วงหน้าแล้ว ห้ามแก้ข้อความเกินการเติมช่องว่าง
2. แจ้ง C ผ่าน LINE พร้อมกัน (ส่งก่อน-แจ้งพร้อมกันได้ ไม่ต้องรอ)
3. ข้อความ **อัปเดตระหว่างแก้** และ **สรุปปิดเหตุ+ขอโทษ** ต้องผ่าน C ก่อนส่งเสมอ (โดยเฉพาะถ้ามีการชดเชย)

## 4. ขั้นตอนมาตรฐาน (Standard Flow)

```
[1] รับเรื่อง/รู้กำหนดการ
      ↓
[2] เปิดเทมเพลตตามประเภท → เติมช่อง {{...}} ให้ครบ
      ↓
[3] เปิด 01-Channel-Matrix → เลือกช่องทาง + จังหวะเวลา ตามแบรนด์และ Tier
      ↓
[4] ไล่ 20-Checklist-ก่อนส่ง ให้ครบทุกข้อ
      ↓
[5] ส่งร่างให้ C อนุมัติ (ตาม Tier ข้อ 3) — C ตอบ "อนุมัติ" เป็นลายลักษณ์อักษรใน LINE/กลุ่ม
      ↓
[6] ส่งจริงตามช่องทางและเวลาที่กำหนด
      ↓
[7] ลง 21-Log-ประกาศ ทันทีหลังส่ง
      ↓
[8] ติดตามผล: เฝ้า inbox/กลุ่มลูกค้า 1 ชม.แรกหลังส่ง — มีคำถามซ้ำ ๆ ให้รวบรวมตอบใน FAQ เดียวกัน
```

## 5. กติกาเนื้อหา (Content Rules)

- ใช้เสียงแอดมินสุภาพลงท้าย "ค่ะ" ตามมาตรฐานเดิมของทีม (อ้างอิง [[08-Message-Templates]])
- บอก **ผลกระทบต่อลูกค้า** ก่อนรายละเอียดเทคนิคเสมอ — ลูกค้าไม่สนว่า server อะไร สนว่า "ใช้อะไรไม่ได้ ตั้งแต่กี่โมง ถึงกี่โมง"
- ระบุ วัน/เดือน/ปี + เวลา แบบเต็มเสมอ เช่น "30 มิ.ย. 2569 เวลา 23:30–01:00 น." ห้ามใช้ "คืนนี้" "พรุ่งนี้"
- แจ้งเปลี่ยนราคา/แพ็กเกจ ต้องส่งล่วงหน้า **≥ 30 วัน** ก่อนมีผล
- ทุกข้อความต้องมีช่องทางติดต่อกลับ (LINE OA / เบอร์ / อีเมลของแบรนด์นั้น)
- ห้ามระบุข้อมูลภายใน: ชื่อพนักงาน, สาเหตุเชิงลึกที่ยังไม่ยืนยัน, ชื่อ vendor/ธนาคารคู่กรณี

## 6. PDPA และข้อกำกับ

- ส่งได้เฉพาะลูกค้าที่มีความสัมพันธ์ทางบริการอยู่ หรือ opt-in รับข่าวสารแล้วเท่านั้น
- **SMS การตลาด (T3)** ต้องมีข้อความทางออก เช่น "ยกเลิกรับข่าวสาร พิมพ์ STOP" ตามแนว NBTC — SMS แจ้งบริการ (T1/T2) ไม่ต้องมี
- ห้ามเปิดเผยข้อมูลส่วนบุคคลของลูกค้ารายใดในประกาศ (อ้างอิงแนวทาง PDPA ที่ C ประกาศ 17 มิ.ย. 2569)
- รายละเอียดเพิ่มเติม: [[PDPA]]

## 7. ตัวชี้วัด (KPI)

| KPI | เป้าหมาย |
|---|---|
| T1: เวลาจากรู้เหตุ → ข้อความ "รับทราบปัญหา" ถึงลูกค้า | ≤ 30 นาที |
| T2: แจ้งลูกค้าล่วงหน้าก่อนปิดปรับปรุง | ≥ 3 วัน |
| ประกาศที่ลง log ครบ | 100% |
| ประกาศที่ผ่านอนุมัติก่อนส่ง (ยกเว้น T1 จังหวะแรก) | 100% |

## 8. การทบทวน

- ทบทวน SOP + เทมเพลตทุก 6 เดือน หรือเมื่อมีแบรนด์/ช่องทางใหม่
- สมาชิกทีม CS ใหม่ต้องอ่านเอกสารชุดนี้ + ทดลองเติมเทมเพลต 1 ครั้งภายใน 7 วันแรก
