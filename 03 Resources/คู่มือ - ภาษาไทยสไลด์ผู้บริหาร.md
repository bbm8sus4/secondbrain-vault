# 🗺️ คู่มือ — ภาษาไทยสำหรับสไลด์/รายงานผู้บริหาร

หลักเลือกคำตอนเขียนสไลด์หรือรายงานภาษาไทยให้บอร์ด/ผู้บริหาร — กลั่นจาก aidebate 3 โมเดล อย่าเขียนภาษาราชการ อย่าเขียนภาษาแปล

> โทนเป้าหมาย: **ภาษาห้องประชุมจริง** ไม่ใช่ภาษาราชการ ไม่ใช่ภาษาแปลจากอังกฤษ — เก็บคำเหมือน "เด้งจากจุดต่ำ", "เร่งเครื่อง", "ก้อนใหญ่สุด", "ห้ามขึ้นบอร์ดก่อนยืนยัน" ไว้ ดันให้ทางการคือผิดทาง

## กฎทอง: คำนามเทคนิคคงอังกฤษได้ — แต่กริยาต้องเป็นไทย

นี่คือกฎเดียวที่กำจัด "กลิ่นภาษาแปล" ได้เกินครึ่ง

| Before (กริยาอังกฤษ) | After (กริยาไทย) |
|---|---|
| Validate spike พ.ค. | ตรวจยืนยันยอดพุ่ง พ.ค. |
| Map customer concentration | เช็คการกระจุกตัวของลูกค้า |
| Double-down API | ทุ่มทรัพยากรให้ API |
| Re-forecast | ปรับประมาณการใหม่ |
| จะ defend เป้าอย่างไร | จะรักษาเป้าอย่างไร |
| reconcile กับต้นทาง | กระทบยอดกับต้นทาง |
| ดัน vector ที่โตชัดสุด | ดันตัวที่โตแรงสุด |
| trajectory ชัดมาก | ทิศทางขาขึ้นชัดมาก |
| kill/reprice | เลิกขาย/ปรับราคา |

## คำที่ส่งสัญญาณผิด — แก้ก่อนทุกอย่าง

| Before | After | ทำไม |
|---|---|---|
| Spike น่าสงสัย | ยอดพุ่ง ยังต้องยืนยันที่มา | "น่าสงสัย" ในห้องประชุมไทยชวนคิดเรื่อง**ทุจริต** ทั้งที่ตั้งใจสื่อแค่ "อาจไม่เกิดซ้ำ" |
| ความเสี่ยง**รวมศูนย์** | ความเสี่ยง**กระจุกตัว** | สลับศัพท์กลางเด็คฟังเป็นคนละเรื่อง — ใช้คำเดียวทั้งเด็ค (ตรงกับ Concentration Risk) |
| ค่าปรับ churn | ค่าปรับยกเลิกสัญญาก่อนกำหนด | churn คือปรากฏการณ์ เก็บเงินจากมันตรงๆ ไม่ได้ — ที่หมายถึงคือ early-termination fee |
| ถูก disrupt **ด้วย** API | โดน API (ของเราเอง) **แย่งตลาด** | "ด้วย" คือ passive แข็งแบบอังกฤษ และถ้าคือ self-cannibalization ต้องบอกชัด — เปลี่ยนน้ำหนักการตัดสินใจ |
| ปิด**บัง**ตัวเลข | **ซ่อน**ตัวเลข | "ปิดบัง" สื่อเชิงลบ (จงใจปกปิด) ใช้ "ซ่อน" ตามชื่อปุ่ม UI |
| ห้ามเอาเลข...ไป**สัญญา** | ห้ามเอาเลข...ไป**รับปากบอร์ด** | "สัญญา" กำกวมกับสัญญาเอกสาร · คงโทน "ห้าม" ไว้ — มันคือ guardrail ที่จงใจ อย่าแก้เป็น "ควร" |

## ลักษณนาม — เลือกให้ตรงประเภท

- **ก้อน** เหมาะกับ**เงิน/รายได้** ("ก้อนใหญ่สุด ~54% ของกลุ่ม" ✓ เก็บไว้) แต่**ไม่เหมาะกับสินค้า** ("API 2 ก้อน" ✗ → "API 2 ตัว")
- อย่าเปลี่ยน "ก้อน" เป็น "ผลิตภัณฑ์หลัก" ใน context รายได้ — ทำให้กลายเป็นภาษารายงานราชการทันที

## ศัพท์เทคนิค — คงอังกฤษได้ vs ต้องเสริมไทย

**คงอังกฤษได้เลย** (วงการใช้ทับศัพท์จริง): API, BOT, run-rate, margin, COGS, SLA, churn, upsell, S-curve, YTD, scenario, Bull/Base/Bear, top-line, spike, Cash Cow, flat fee

**ต้องวงเล็บไทยครั้งแรกที่ปรากฏ** (เพราะถ้าคนฟังคนเดียวไม่ชัวร์ ข้อสรุปพังหมด):
- **recurring (รายได้ประจำ) / one-off (รายได้ครั้งเดียว)** — แกนของรายงานรายได้ทุกชุด
- **pivot / harvest** → "(เปลี่ยนทิศ หรือ รีดกำไรจากฐานเดิมโดยหยุดลงทุนเพิ่ม)" — เป็นทางเลือกที่บอร์ดต้องเคาะ ⚠️ harvest **ไม่ใช่** "ยุติการดำเนินงาน" จะพาตัดสินใจผิด
- **unit economics** → "(กำไรต่อหน่วย/ต่อ transaction)" ⚠️ **ห้าม**แปลเป็น "เศรษฐกิจองค์กร" (ผิดความหมายธุรกิจล้วน)
- **PMF** → ครั้งแรกเขียนเต็ม "product-market fit (ตลาดต้องการจริง)"
- **single point of failure** → "(สะดุดจุดเดียว กระทบทั้งกลุ่ม)"
- **flat fee** → "(เหมาจ่าย)"

## ตัวเลขในข้อความ — ระบุฐานคำนวณ เสมอ

ห้ามเขียน "Run-rate ×12 ≈ ฿122M" ลอยๆ — บอร์ดจะคิดตามแล้วเจอ 11.5×12=138M ≠ 122M แล้วตั้งคำถามกับ**ทั้งเด็ค**
ที่ถูก: "**เฉลี่ย 5 เดือน** ×12 ≈ ฿122M (run-rate)"

หลักรวบยอด: ทุกตัวเลข derived ต้องบอกว่ามาจากฐานไหน ("เฉลี่ย 5 เดือน", "ยืนระดับ พ.ค.", "เทียบ พ.ค.") — กันบอร์ดจับผิดเลขแล้วเสียความเชื่อถือทั้งงาน

## หลักรวบยอด (memorize)

1. **คำนามเทคนิคคงอังกฤษได้ — แต่กริยาต้องเป็นไทย** (กฎเดียวที่กำจัดกลิ่นแปล 50%)
2. **เลี่ยงสัญญาณทุจริต** ("น่าสงสัย") และ**ใช้ศัพท์เดียวทั้งเด็ค** ("กระจุกตัว" ไม่ใช่ "รวมศูนย์")
3. **อย่าดันไปทางราชการ** — "ก้อนใหญ่สุด" / "ห้ามขึ้นบอร์ด" คือจุดแข็ง ไม่ใช่ปัญหา
4. **ทุกตัวเลข derived ต้องมีฐานคำนวณ** — กันบอร์ดจับผิด
