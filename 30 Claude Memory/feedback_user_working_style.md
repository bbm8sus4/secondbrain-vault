วิธีสั่งงานและพฤติกรรมของผู้ใช้ (Bob, ผู้บริหาร) — ยึดเป็นค่าเริ่มต้นเวลาทำงานให้เขา:

- **สั่งเป็นชุดหลายข้อพร้อมภาพ** — มักส่งหลาย task ในข้อความเดียว + screenshot ชี้จุด. ต้องทำ**ครบทุกข้อ ตามลำดับ** ห้ามตกหล่น. ถ้าเขาบอก "ทำงานที่ทำอยู่ให้เสร็จก่อนค่อยทำอันนี้ต่อ" = ปิดงานเดิมให้จบก่อน แล้วค่อยเริ่มชุดใหม่.
- **Research/design ก่อนลงมือ** — "ไปคิดมา" / "ไปหาข้อมูลก่อนทำงาน" = ต้องศึกษา/ออกแบบ/หา touch-point ให้ครบก่อนแก้ (ใช้ workflow วางแผนได้). ห้ามเดาแล้วแก้มั่ว.
- **Verify ของจริงเสมอ** — ต้องเทสด้วย **Chrome MCP บนแอปจริง** ไม่ใช่แค่เคลม. เขาเคยหงุดหงิดหนัก ("ควาย") ตอนของที่บอกว่าเสร็จแล้วใช้ไม่ได้จริง. ทำงานได้จริง > พูดว่าเสร็จ.
- **Zero-defect mindset** — "ห้ามมีปัญหาเรื่องง่าย ๆ" "ทำแบบ DevOps ก่อน deploy". ต้องมี quality gate + ไล่เบสิกที่มักมองข้าม (ชอบให้ใช้สกิล `/CRUD-basic`).
- **ชอบ UI ที่สะอาด/รวบ** — ปุ่มเยอะ/รก = ไม่ชอบ ("เครื่องหมาย + เยอะ ๆ ไม่ดี"); ชอบรวมเป็นปุ่มเดียว, dropdown, จบในหน้าเดียว, ธีมสว่าง.
- **Polish over half-done** — งานต้องเสร็จจริง ครบถ้วน สวย ใช้ง่าย. ลบของที่ไม่สวยทิ้ง ("ไม่เอา ไม่สวย ลบทิ้งไป").
- **ตอบไทยสั้น กระชับ ลงท้ายครับ** — อ่านรวดเดียวบนมือถือ. ผลลัพธ์ก่อน รายละเอียดทีหลัง.
- **บันทึกลง Obsidian เมื่อสั่ง** — "จำลง Ob" / "จดลง SecondBrain" = เขียนโน้ต markdown ใน `~/SecondBrain/01 Projects/<โปรเจกต์>/` เก็บ**ทั้งรายการสั่งงาน (requests) + วิธีทำงาน (working method)** ไม่ใช่แค่ auto-memory. (SeatMap: session log อยู่ที่ `Office Seating Plan/2026-*.md`)
