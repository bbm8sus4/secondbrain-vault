For automated mass-send loops (LINE OA, Telegram, batch DMs) across many recipients: pace SLOWLY. Don't optimize for total wall-clock speed.

**Why:** During the EasyBot Affiliate broadcast (1,243 LINE OA chats, 2026-05-06), at ~7.5s/chat pace, duplicate text/image sends started appearing across chats — likely from modal dialogs not closing in time before the next chat began, leaving stale UI state that bled into the next iteration. User stopped the loop and explicitly said "อย่าทำเร็วเกินไป เพราะข้อความหรือรูปภาพที่ส่งซ้ำกัน ช้าลงหน่อย ค่อย ๆ ทำ".

**How to apply:**
- Default: 2x the timing that "works in test" — tests run on a clean state, real loops accumulate state.
- After every UI action that triggers a modal/dialog, wait for it to FULLY close before proceeding (poll `document.querySelector('.modal.d-block')` is null), and add a buffer (1–2s extra).
- Between recipients: minimum 3–5s throttle, more if uploads are involved.
- Image/file uploads: assume up to 10s per upload; modal_did_not_close is a real failure mode that re-running won't fix.
- Always include modal-cleanup at start of each iteration (close any leftover dialog before navigating to next).
- Snapshot progress to disk every N items (e.g. every 20) so a tab crash doesn't lose state.
- Before resuming after a crash: SCAN the recipient list for already-sent markers (e.g. last-message preview contains broadcast text) — never trust in-memory state alone.
